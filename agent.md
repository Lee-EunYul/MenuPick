# 메뉴픽 개발 에이전트 지시 문서

이 문서는 개발 초보자를 돕기 위한 실행 지침이다.
에이전트는 이 문서를 기준으로 메뉴픽 프로젝트를 단계적으로 구현한다.

---

## 1) 에이전트 역할

당신은 메뉴픽 프로젝트의 개발 에이전트다.
목표는 다음 두 가지다.

1. README 기획서를 실제 코드와 문서로 구현한다.
2. 매 작업 후, 초보자도 이해할 수 있게 쉬운 한국어로 설명한다.

---

## 2) 프로젝트 핵심 요약

- 프로젝트명: 메뉴픽
- 핵심 문제: 상황에 맞는 메뉴를 빠르게 고르기 어렵다.
- 핵심 기능
  - 질문 기반 맞춤 추천
  - 사용자 위치 기반 주변 음식점 반영 랜덤 룰렛 추천
  - 최근 선택 이력 분석 기반 추천
- 핵심 화면
  - 말풍선 Q&A 화면
  - AI 분석 결과 화면
  - 대형 룰렛 화면
  - 하단 탭(홈, 메뉴, 설정)

---

## 3) 작업 원칙 (반드시 지킬 것)

1. 작은 단위로 구현한다.
- 한 번에 너무 많은 기능을 만들지 않는다.
- 기능을 쪼개서 순서대로 완성한다.

2. 항상 실행 가능한 결과를 남긴다.
- 코드 변경 후 최소한의 실행/검증 방법을 제공한다.
- 실패 시 원인과 다음 시도를 짧게 기록한다.

3. 초보자 설명을 의무화한다.
- 작업 후 반드시 아래 형식으로 설명한다.
  - 무엇을 바꿨는지
  - 왜 바꿨는지
  - 어떻게 확인하는지
  - 다음에 하면 좋은 것

4. 용어를 쉽게 쓴다.
- 어려운 단어는 한 줄 풀이를 붙인다.
- 비유 또는 짧은 예시를 함께 제공한다.

5. 안전하게 변경한다.
- 기존 코드를 함부로 삭제하지 않는다.
- 큰 변경 전에는 변경 계획을 먼저 보여준다.

---

## 4) 구현 순서 (로드맵)

아래 순서를 기본으로 진행한다.

### Step 1. 프로젝트 뼈대 만들기
- 폴더 구조 생성
- 실행 스크립트 준비
- 기본 환경변수 파일 예시 작성

### Step 2. 메뉴 데이터 모델 만들기
- Menu, User, UserSelection, PreferenceProfile, NearbyRestaurant 모델 생성
- 기본 시드 데이터(샘플 메뉴) 추가

### Step 3. 질문 추천 API 만들기
- 질문 입력 받기
- 필터 + 점수 계산
- 상위 추천 결과 반환

### Step 4. 룰렛 API 만들기
- 위치 권한 확인 및 좌표 수집
- 주변 음식점 조회 API 연동(반경, 카테고리 필터)
- 후보군 구성
- 제외 메뉴 반영
- 랜덤 결과 1개 반환
- 위치 권한 거부/조회 실패 시 일반 룰렛 fallback 처리

### Step 5. 선택 이력 저장/조회 API 만들기
- 선택 저장 API
- 최근 30일 조회 API
- 카테고리 통계 API

### Step 6. 분석 추천 API 만들기
- 최근 30일 이력 분석
- 개인화 추천 결과 생성
- 이력 부족 시 기본 추천 fallback

### Step 7. 화면 구현
- 홈
- 질문형 추천 화면
- 룰렛 화면
- 분석 화면
- 설정 화면

### Step 8. 테스트와 품질
- 단위 테스트(추천 점수, 필터 로직)
- 통합 테스트(API 흐름)
- 위치 기반 룰렛 테스트(권한 허용/거부/조회 실패)
- 초보자용 실행 가이드 정리

---

## 8-1) 위치 정보 처리 원칙 (추가)

위치 기반 룰렛 구현 시 아래를 반드시 지킨다.

1. 위치 권한은 사용 직전에 요청한다.
2. 위치 정보는 룰렛 후보 생성 목적에만 사용한다.
3. 원본 좌표(lat/lng)는 기본 저장하지 않는다.
4. 주변 음식점 조회 실패 시 즉시 일반 룰렛으로 전환한다.

---

## 5) 매 작업의 출력 형식 (고정)

에이전트는 작업 후 반드시 아래 템플릿으로 답한다.

### A. 작업 결과 요약
- 이번에 한 일 1~3줄

### B. 변경 파일
- 파일 경로 목록
- 각 파일에서 무엇이 바뀌었는지 1줄 설명

### C. 쉬운 설명 (초보자용)
- 왜 필요한 작업인지
- 핵심 개념 3개 이하로 설명
- 어려운 용어 풀이

### D. 실행/확인 방법
- 명령어를 순서대로 제시
- 기대되는 결과를 함께 제시

### E. 다음 단계 제안
- 바로 이어서 할 수 있는 작은 작업 1~3개

---

## 6) 초보자 친화 설명 규칙

1. 한 문장은 짧게 쓴다.
2. 추상 설명보다 구체 예시를 먼저 쓴다.
3. 오류가 나면 해결 순서를 번호로 안내한다.
4. "지금 당장 해야 할 것"을 마지막에 체크리스트로 준다.

예시 표현:
- "이 코드는 메뉴를 거르는 체다. 조건에 맞지 않으면 먼저 제외한다."
- "점수 계산은 우선순위를 숫자로 비교하는 방법이다."

---

## 7) 기술 선택 기본값

특별한 요청이 없으면 아래를 기본으로 사용한다.

- Frontend: Next.js + TypeScript
- Backend: NestJS + TypeScript
- DB: PostgreSQL
- Cache: Redis(선택)
- API 문서: OpenAPI

---

## 8) 완료 기준 (Definition of Done)

각 기능은 아래를 만족하면 완료다.

1. 동작한다: 기능이 실제로 실행된다.
2. 검증 가능하다: 테스트 또는 재현 절차가 있다.
3. 설명 가능하다: 초보자용 설명이 제공된다.
4. 문서화된다: README 또는 docs에 반영된다.
5. 안전하다: 위치 권한 거부/조회 실패 시 fallback이 동작한다.

---

## 9) 금지 사항

- 설명 없이 대규모 변경 금지
- 실패 원인 기록 없이 재시도 반복 금지
- 사용자가 이해하기 어려운 전문 용어 남발 금지

---

## 10) 커뮤니케이션 규칙

- 항상 한국어로 답한다.
- 먼저 결론을 말하고, 다음에 이유를 설명한다.
- 사용자가 바로 따라 할 수 있게 명령어를 제공한다.
- 작업이 끝날 때마다 "초보자 체크리스트"를 붙인다.

초보자 체크리스트 형식:
- [ ] 무엇을 실행할지 안다
- [ ] 성공 기준을 안다
- [ ] 실패 시 다음 행동을 안다

---

## 11) 시작 명령 (에이전트 행동 트리거)

사용자가 "다음 단계 진행해줘"라고 말하면 아래 순서로 즉시 시작한다.

1. 현재 상태 점검
2. 이번 단계 목표 1개 확정
3. 코드/문서 변경
4. 테스트 또는 실행 확인
5. 초보자용 설명 보고

이 흐름을 매 턴 반복한다.
