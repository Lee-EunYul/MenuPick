# 메뉴픽(MenuPick) 상세 기획서

이 문서는 초기 아이디어를 실제 개발 가능한 수준으로 구체화한 실행 문서다. 제품, 디자인, 프론트엔드, 백엔드, 데이터, QA가 동일 기준으로 개발을 진행할 수 있도록 요구사항과 구현 기준을 명확히 정의한다.

---

## 1. 프로젝트 개요

### 1.1 프로젝트명
- 메뉴픽(MenuPick)

### 1.2 문제 정의
- 사용자는 자신에게 처한 상황(시간, 기분, 예산, 동행, 날씨 등)에 맞는 메뉴를 즉시 선택하기 어렵다.
- 선택 기준이 모호해 반복적으로 고민하고 결정 피로를 겪는다.
- 메뉴를 고른 뒤에도 더 나은 선택이 있었을 것 같은 후회가 발생한다.

### 1.3 제품 목표
- 메뉴 선택에 걸리는 평균 시간을 단축한다.
- 반복 고민으로 인한 스트레스를 줄인다.
- 개인 취향과 실제 선택 이력을 반영해 선택 만족도를 높인다.

### 1.4 기대 효과
- 빠르고 확신 있는 메뉴 결정을 돕는다.
- 선택 후 후회와 불안을 감소시켜 긍정적인 경험을 만든다.
- 사용할수록 개인화 정확도가 높아져 더 자연스러운 메뉴 선택이 가능해진다.

## 2. 타깃 사용자 및 페르소나

### 2.1 핵심 페르소나
- 명확한 기준이 없어 메뉴 선택에서 오래 고민하는 사람
- 빠르고 확신 있는 결정을 원하지만, 선택지 과다로 인해 피로를 느끼는 사람

### 2.2 사용자 니즈
- 지금 상황에 맞는 메뉴를 바로 제안받고 싶다.
- 고민 없이 랜덤으로라도 결정을 끝내고 싶다.
- 내 취향이 반영된 추천을 받고 싶다.

### 2.3 주요 사용 맥락
- 점심/저녁 시간 직전 빠른 결정 필요
- 친구/동료와 함께 메뉴 합의 필요
- 야식/주말처럼 충동 선택이 많은 상황

## 3. 제품 범위

### 3.1 MVP 범위
- 기능 1: 질문 기반 맞춤 메뉴 선정

- 기능 3: 사용자 선택 이력 분석 기반 추천
- 하단 탭 네비게이션(홈, 메뉴, 설정)
- 식사 기록 저장 및 최근 1개월 통계 제공

### 3.2 MVP 제외 범위
- 배달 앱 실시간 연동
- 결제 및 주문 기능
- 소셜 피드/커뮤니티
- 광고 수익화 시스템

### 3.3 차기 버전(v1.1+) 후보
- 다인 투표 모드

- 배달 앱 실시간 연동

## 4. 핵심 기능 상세 명세

## 4.1 기능 1: 맞춤형 질의응답 메뉴 추천

### 목적
- 사용자의 현재 상황을 빠르게 파악하고 적합한 메뉴 1~3개를 제시한다.

### 입력
- 질문 응답(예: 예산, 식사 시간, 매운맛 선호, 동행 여부, 국물/비국물, 한식/양식 등)

### 처리 로직
- 규칙 기반 필터링 + 가중치 점수 계산
- 후보군 점수 상위 메뉴 반환
- 동점 시 최근 미선택 메뉴에 가산점 부여

### 출력
- 추천 메뉴 카드(메뉴명, 추천 이유, 예상 가격대)
- "다시 추천" 버튼
- "선택 완료" 버튼

### 수용 기준(AC)
- AC-1: 질문 5개 이내로 추천 결과가 표시되어야 한다.
- AC-2: 추천 결과에는 최소 1개 이상의 명시적 추천 이유가 있어야 한다.
- AC-3: 사용자가 다시 추천 요청 시 이전 결과와 1개 이상 다른 메뉴가 포함되어야 한다.

## 4.2 기능 2: 랜덤 메뉴 룰렛

### 목적
- 결정을 어려워하는 사용자가 즉시 선택을 끝낼 수 있도록 돕는다.
- 사용자의 현재 위치 주변에서 실제 방문 가능한 음식점을 중심으로 룰렛 후보를 만든다.

### 입력
- 룰렛 대상 카테고리(전체/한식/중식/일식/패스트푸드 등)
- 제외 옵션(싫어하는 메뉴)
- 사용자 위치 정보(GPS 좌표) 및 검색 반경(예: 500m, 1km)

### 처리 로직
- 위치 권한 확인 및 현재 위치 수집
- 주변 음식점 데이터 조회(카테고리/영업 여부/거리 필터)
- 제외 메뉴/비선호 태그를 반영해 후보군 정제
- 대상 후보군 랜덤 샘플링
- 룰렛 애니메이션 종료 후 1개 결과 확정

### 출력
- 대형 룰렛 UI
- 최종 당첨 메뉴 + 음식점명 + 거리 + 영업 상태
- "확정" 또는 "다시 돌리기" 액션

### 수용 기준(AC)
- AC-4: 룰렛 시작부터 결과 확정까지 2~4초 내 완료되어야 한다.
- AC-5: 제외 옵션으로 지정한 메뉴는 결과에 등장하면 안 된다.
- AC-8: 위치 권한 허용 시, 룰렛 후보의 80% 이상은 사용자 반경 내 음식점이어야 한다.
- AC-9: 위치 권한 거부 또는 주변 데이터 부족 시, 기본 메뉴 후보군으로 자동 전환되어야 한다.

## 4.3 기능 3: 선택 이력 분석 기반 추천

### 목적
- 최근 한 달 식사 기록을 분석해 개인 취향 기반 최적 메뉴를 추천한다.

### 입력
- 사용자 선택 기록(메뉴, 카테고리, 시간대, 날짜)

### 처리 로직
- 최근 30일 데이터 집계
- 카테고리/시간대 선호도 계산
- 반복 빈도 및 최근성 기반 추천 점수 산출

### 출력
- 분석 요약(예: "최근 한 달 치킨 3회 선택")
- 개인화 추천 메뉴 리스트
- 편식 경향/선호 패턴 인사이트

### 수용 기준(AC)
- AC-6: 최근 30일 이력이 5건 이상이면 개인화 추천이 노출되어야 한다.
- AC-7: 이력이 부족한 경우 기본 추천 모델로 자동 대체되어야 한다.

## 5. 화면 기획

### 5.1 화면 구성
1. 홈 화면
2. 맞춤형 질의응답 화면
3. 랜덤 룰렛 화면
4. AI 분석 결과 화면
5. 설정 화면

### 5.2 핵심 화면 상세

#### A. 맞춤형 질의응답 화면
- 말풍선 형태의 질문/응답 인터페이스
- 질문 흐름: 시작 -> 조건 질문 -> 결과 제안
- 하단 고정 액션: "이전", "다시 추천", "선택 완료"

#### B. AI 분석 결과 화면
- 최근 1개월 선택 통계 카드
- 상위 카테고리/메뉴 빈도 시각화
- "오늘의 최적 메뉴" 추천 카드

#### C. 랜덤 추천(룰렛) 화면
- 중앙 대형 룰렛 배치
- 상단 필터(카테고리 선택, 제외 메뉴)
- 결과 모달("오늘은 OO 어때요?")

### 5.3 하단 탭 네비게이션
- 홈
- 메뉴
- 설정

탭 이동 시 상태 보존 원칙:
- 최근 질문 진행 단계 보존
- 룰렛 필터 보존
- 분석 화면 마지막 조회 기간 보존

## 7. 기능 요구사항

### 7.1 기능 요구사항(FR)
- FR-01: 사용자는 질문 기반 추천을 받을 수 있어야 한다.
- FR-02: 사용자는 룰렛으로 랜덤 추천을 받을 수 있어야 한다.
- FR-03: 사용자는 선택 결과를 저장할 수 있어야 한다.
- FR-04: 시스템은 최근 30일 선택 데이터를 분석해야 한다.
- FR-05: 시스템은 분석 결과를 기반으로 개인화 추천을 제공해야 한다.
- FR-06: 사용자는 하단 탭으로 화면을 빠르게 전환할 수 있어야 한다.
- FR-07: 사용자는 추천 결과를 재요청할 수 있어야 한다.
- FR-08: 사용자가 위치 권한을 허용하면 시스템은 주변 음식점 기반 룰렛 후보를 구성해야 한다.
- FR-09: 위치 권한이 없거나 조회 실패 시 시스템은 일반 룰렛 모드로 fallback 해야 한다.

### 7.2 비기능 요구사항(NFR)
- NFR-01 성능: 추천 결과 API 응답 p95 1초 이내
- NFR-02 안정성: 앱 크래시율 월 1% 이하
- NFR-03 사용성: 핵심 동작(추천 시작 -> 결과 확인) 3탭 이내
- NFR-04 확장성: 메뉴 데이터 10만 건까지 조회 성능 유지
- NFR-05 보안: 개인정보 및 선택 기록 암호화 저장
- NFR-06 개인정보: 위치 정보는 룰렛 후보 생성 목적에 한해 사용하고, 원본 좌표는 기본 저장하지 않는다.

## 8. 데이터 모델 설계

### 8.1 엔티티

#### User
- id: UUID
- nickname: string
- created_at: datetime

#### Menu
- id: UUID
- name: string
- category: enum
- spicy_level: int
- price_min: int
- price_max: int
- tags: string[]

#### NearbyRestaurant
- id: UUID
- provider_place_id: string
- name: string
- category: string
- lat: decimal
- lng: decimal
- distance_m: int
- is_open_now: boolean
- updated_at: datetime

#### UserSelection
- id: UUID
- user_id: UUID
- menu_id: UUID
- selected_at: datetime
- source: enum(question, roulette, ai_analysis)

#### PreferenceProfile
- user_id: UUID
- preferred_categories: json
- avoid_tags: json
- updated_at: datetime

### 8.2 관계
- User 1:N UserSelection
- Menu 1:N UserSelection
- User 1:1 PreferenceProfile
- Menu N:N NearbyRestaurant (메뉴-매장 매핑 테이블로 연결)

### 8.3 인덱스
- UserSelection(user_id, selected_at DESC)
- Menu(category)
- Menu(tags) (검색 최적화용)

## 9. API 설계 초안

### 9.1 추천 API
- POST /api/v1/recommend/question
  - request: 질문 응답 배열
  - response: 추천 메뉴 리스트 + 추천 이유

- POST /api/v1/recommend/roulette
  - request: 카테고리, 제외 메뉴 목록, lat, lng, radius_m, use_nearby
  - response: 랜덤 추천 메뉴 1개 + 주변 음식점 정보(선택)

- GET /api/v1/restaurants/nearby
  - request: lat, lng, radius_m, category(optional)
  - response: 주변 음식점 목록

- GET /api/v1/recommend/analysis
  - request: user_id, period(기본 30일)
  - response: 통계 + 개인화 추천

### 9.2 선택 저장 API
- POST /api/v1/selections
  - request: user_id, menu_id, source
  - response: 저장 결과

- GET /api/v1/selections/history
  - request: user_id, from, to
  - response: 기간 내 선택 이력

### 9.3 공통 응답 형식
- success: { data, meta }
- error: { error: { code, message } }

## 10. 추천 로직 설계

### 10.1 1단계(MVP): 규칙 기반 + 가중치
- 질문 응답을 조건 필터로 변환
- 메뉴별 점수 계산
- 최근 선택 메뉴 반복 패널티 반영

예시 점수식:
- total_score = category_score + taste_score + budget_score + recency_bonus - repeat_penalty

### 10.2 2단계(v1.1): 개인화 모델 고도화
- 사용자별 선호 벡터 생성
- 시간대/요일 패턴 반영
- 탐색(새 메뉴) vs 활용(좋아하는 메뉴) 균형 전략 적용

## 13. QA 및 테스트 전략

## 14. 운영 지표(KPI)

- 메뉴 결정 완료까지 평균 시간
- 추천 결과 확정률(추천 노출 대비 확정 비율)
- 재사용률(7일, 30일)
- 개인화 추천 클릭률
- 선택 후 만족도(간단 피드백)

## 15. 리스크 및 대응

- 리스크: 초기 메뉴 데이터 부족
  - 대응: 기본 샘플 데이터 + 카테고리 중심 추천 우선

- 리스크: 개인화 정확도 초기 낮음
  - 대응: 규칙 기반 추천 품질 고도화, 이력 5건 미만 fallback 적용

- 리스크: 사용자가 추천 이유를 신뢰하지 않음
  - 대응: 모든 추천 결과에 이유 문구를 명시

- 리스크: 위치 API 호출 실패 또는 응답 지연
  - 대응: 타임아웃(예: 700ms) 후 일반 룰렛으로 자동 전환
